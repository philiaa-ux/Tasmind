# AI 챗봇 기반 Tasmota 자동화 시스템 사용 매뉴얼

본 매뉴얼은 채팅형 AI 에이전트와의 대화만으로 스마트 기기들을 자동화하는 방법을 안내합니다.  
**특정 장치 이름은 `[기기]`와 같은 범용 표현으로 대체**하여, 누구나 자신의 환경에 맞춰 적용할 수 있도록 작성되었습니다.

---

## 📌 1. 시스템 개요

TasMind의 자동화 시스템은 **두 가지 유형의 규칙**을 지원합니다.

| 규칙 유형 | 실행 위치 | 특징 | 적합한 사용 사례 |
|-----------|-----------|------|-------------------|
| **시스템 규칙 (Server Rule)** | TasMind 백엔드 서버 | 중앙 집중식 관리, 복잡한 조건/다중 장치 연동 가능, 로그 기록 보관 | 스마트홈/팜/빌딩/팩토리 전체 자동화 |
| **물리 규칙 (Physical Rule)** | Tasmota 장치 자체 | 장치 내부에서 독립 실행, 서버 의존성 없음, 초고속 응답(ms 단위) | 인터넷 단절 대비, 단일 장치 긴급 제어 |

> 💡 **핵심 개념**:  
> - **시스템 규칙**: 서버에서 실행되며, 여러 장치 간 복잡한 조건과 연동이 가능합니다.  
> - **물리 규칙**: Tasmota 장치에 직접 저장되어 실행됩니다. **"Tasmota Rule"** 또는 **"물리 룰"** 이라고도 불립니다.

자세한 규칙 유형별 특성은 [2.2절 규칙 유형 비교](#22-규칙-유형-비교)에서 확인하세요.

---

## 🧩 2. 기본 개념 (식별자 규칙)

AI와 대화할 때는 아래 규칙에 따라 장치를 지칭하면 정확하게 인식합니다.

| 구성 요소 | 예시 | 설명 |
|:---|:---|:---|
| **장치 ID** | `[기기_ID]` | Tasmota 장치의 고유 이름 (예: `tasmota_거실`) |
| **스위치 채널** | `[기기_ID]_POWER[번호]` | 릴레이 채널 (예: `POWER1`, `POWER2`) |
| **센서 값** | `[기기_ID]_[센서명]` | 온습도(AM2301), 조도(BH1750), pH 등 |

> 💡 **팁**: 장치에 별명(닉네임)을 지정하면 `[별명]`으로도 호출할 수 있습니다.  
> (예: `"거실 온도 센서 값 알려줘"`)

## ⚡ 2. 규칙 유형 비교

AI에게 자동화 규칙을 요청할 때는 **규칙이 어디에서 실행될지**를 함께 명시해야 합니다.

| 구분 | 시스템 규칙 (Server Rule) | 물리 규칙 (Physical Rule) |
|------|---------------------------|---------------------------|
| **실행 위치** | TasMind 백엔드 서버 | Tasmota 장치 내부 |
| **응답 속도** | 네트워크 지연 영향 (보통 1~3초) | 즉시 실행 (ms 단위) |
| **인터넷 의존성** | 인터넷 연결 필수 | 인터넷 없이도 동작 (로컬) |
| **복잡한 조건** | 여러 장치 간 AND/OR/시간 조건 지원 | 단일 장치 내 기본 조건만 지원 |
| **로그/추적** | 모든 실행 이력 저장 | 장치 재부팅 시 로그 초기화 |
| **생성 가능 개수** | Tier에 따라 제한 (최대 100개) | 장치당 최대 16개 (Rule1~Rule16) |

### 🎯 규칙 유형 선택 가이드

| 상황 | 추천 규칙 유형 |
|------|----------------|
| 여러 장치를 함께 제어해야 할 때 | **시스템 규칙** |
| 인터넷이 불안정한 환경 | **물리 규칙** |
| 밀리초 단위의 빠른 응답이 필요할 때 | **물리 규칙** |
| 복잡한 조건 (예: 시간대 + 센서값 조합) | **시스템 규칙** |
| 단일 장치의 간단한 ON/OFF 자동화 | **물리 규칙** (더 간단하고 빠름) |

## 🔧 3. 시나리오(템플릿) 생성 및 적용

### 3.1 시나리오 등록 (자연어 설명)

원하는 자동화 로직을 **자연어**로 설명하면, AI가 내부적으로 DSL(JSON) 구조를 생성하여 시나리오 템플릿으로 등록합니다.

> **👤 사용자 입력 예시**
> ```
> 스마트홈 도메인에 새로운 자동화 시나리오 템플릿을 등록해줘.
> 
> 시나리오 이름: 온습도_환기_제어
> 시나리오 설명: 온도와 습도에 따라 환풍기와 조명을 자동 제어
> 필요 장치 역할: temperature_sensor, humidity_sensor, living_room_light, fan_actuator
> 사용할 기본 파라미터: temp_max (기본값: 28), humi_max (기본값: 60)
> 
> 동작 규칙:
>   조건: 온도 >= temp_max AND 습도 <= humi_max
>   동작: living_room_light 켜기(ON), fan_actuator 켜기(ON)
>   조건: 온도 < temp_max OR 습도 > humi_max
>   동작: living_room_light 끄기(OFF), fan_actuator 끄기(OFF)
> 
> 실행 모드: toggle
> ```

> **🤖 AI 응답**
> ```
> ✅ 시나리오 템플릿이 성공적으로 등록되었습니다!
> 시나리오 ID: sc_20250703_a1b2c3
> 실행 모드: toggle
> 생성된 규칙 유형: 시스템 규칙 (서버 기반)
> 
> 💡 Tip: 장치에 직접 설치하려면 "물리 규칙으로 변환해줘"라고 말씀해주세요.
> ```

---

### 3.2 시나리오 적용 (장치 바인딩)

등록된 시나리오에 실제 내 장치들을 연결(바인딩)하여 활성화합니다.

> **👤 사용자 입력 예시**
> ```
> 방금 등록된 시나리오 sc_20250703_a1b2c3를 내 환경에 적용해줘.
> 
> 장치 바인딩:
>   temperature_sensor: 거실_온도센서
>   humidity_sensor: 거실_습도센서
>   living_room_light: 거실_조명_POWER1
>   fan_actuator: 환풍기_POWER1
> 
> 파라미터: temp_max=28, humi_max=60
> 쿨다운: 60초
> 규칙 유형: 시스템 규칙으로 적용해줘
> ```

> **🤖 AI 응답**
> ```
> ✅ 시나리오 적용 완료! (시스템 규칙)
> 생성된 규칙: 2개
>   - 규칙 #1: 온도 >= 28°C AND 습도 <= 60% → 조명 ON, 환풍기 ON
>   - 규칙 #2: 온도 < 28°C OR 습도 > 60% → 조명 OFF, 환풍기 OFF
> 각 장치에 Rule이 배포되어 즉시 자동화가 시작됩니다.
> 
> ⚠️ 참고: 시스템 규칙은 서버가 온라인 상태일 때만 동작합니다.
> ```

---

### 3.3 물리 규칙으로 변환 (장치 직접 설치)

시스템 규칙으로 생성된 자동화를 **Tasmota 장치 내부에 직접 설치**하려면 변환을 요청할 수 있습니다.

> **👤 사용자 입력 예시**
> ```
> 방금 적용한 시나리오를 물리 규칙으로 변환해서 장치에 직접 설치해줘.
> 대상 장치: 거실_온도센서 (온도/습도 센서가 함께 있는 장치)
> ```

> **🤖 AI 응답**
> ```
> ✅ 물리 규칙 변환 완료!
> 
> [거실_온도센서] 장치에 설치할 Rule:
> 
> Rule1: ON tele#temperature>=28 AND tele#humidity<=60 DO POWER1 ON ENDON
> Rule2: ON tele#temperature<28 OR tele#humidity>60 DO POWER1 OFF ENDON
> 
> 설치 방법:
> 1. [거실_온도센서] 장치의 웹 콘솔(Tasmota Console)에 접속
> 2. 아래 명령어를 순서대로 입력:
> 
> Rule1 1
> Rule2 2
> Rule1 ON
> Rule2 ON
> 
> ⚠️ 주의: 물리 규칙은 장치 자체에서 실행되므로, 서버와 무관하게 동작합니다.
> 장치가 인터넷과 단절되어도 정상 작동합니다.
> ```

## ⚡ 4. 개별 규칙 생성 (시나리오 없이)

간단한 조건-동작은 시나리오 템플릿 없이 **개별 규칙**으로 바로 생성할 수 있습니다.

### 4.1 시스템 규칙 (서버 기반)

> **👤 사용자 입력 예시**
> ```
> 시스템 규칙 하나 생성해줘.
> 규칙 이름: 온도_30도_초과_에어컨_ON
> 실행 모드: 반복 실행
> 쿨다운: 5분
> 조건: 거실_온도센서의 온도가 30도를 초과하면
> 동작: 에어컨_POWER1을 ON으로 변경
> ```

> **🤖 AI 응답**
> ```
> ✅ 시스템 규칙이 생성되었습니다!
> 규칙 ID: rule_20250703_x9y8z7
> 조건: 거실_온도센서.temperature > 30
> 동작: 에어컨.POWER1 ON
> 실행 모드: repeat (5분 쿨다운)
> 상태: 활성화
> ```

---

### 4.2 물리 규칙 (Tasmota 로컬 Rule)

서버 의존성을 줄이고 더 빠른 응답이 필요할 때는 장치 자체에 물리 규칙을 설정할 수 있습니다.

> **👤 사용자 입력 예시**
> ```
> 환풍기 장치에 물리 규칙을 설정해줘.
> 조건: 온도 센서값이 28도 이상이고 습도가 65% 이상이면
> 동작: 환풍기_POWER1을 ON
> 실행 모드: 반복 (조건이 유지되는 동안 계속 ON)
> ```

> **🤖 AI 응답**
> ```
> ✅ 물리 규칙 설정 완료!
> 
> [환풍기] 장치에 설치할 Rule:
> 
> Rule1: ON tele#temperature>=28 AND tele#humidity>=65 DO POWER1 ON ENDON
> 
> 설치 명령어 (Tasmota Console):
> > Rule1 1
> > Rule1 ON
> 
> 💡 물리 규칙은 장치가 재부팅되어도 유지됩니다.
> 삭제하려면 "Rule1 OFF" 후 "Rule1 0"을 입력하세요.
> ```

---

### 4.3 규칙 유형 자동 선택 (AI 추천)

규칙 유형을 명시하지 않으면 AI가 상황에 맞게 자동으로 추천합니다.

> **👤 사용자 입력 예시**
> ```
> 온도가 25도 이상이면 선풍기를 켜는 규칙 만들어줘.
> (규칙 유형을 명시하지 않음)
> ```

> **🤖 AI 응답**
> ```
> ✅ 규칙 생성 완료! (AI가 자동으로 물리 규칙을 추천합니다)
> 
> 💡 추천: 이 규칙은 단일 장치의 간단한 조건-동작이므로 물리 규칙이 적합합니다.
> 
> [선풍기] 장치에 설치할 물리 규칙:
> Rule1: ON tele#temperature>=25 DO POWER1 ON ENDON
> 
> ⚠️ 만약 서버 기반 규칙이 필요하시면 "시스템 규칙으로 변환해줘"라고 말씀해주세요.
> ```

## 🏠 5. 가상 노드(Virtual Node) 관리

여러 장치를 하나의 논리적 공간(방, 구역, 존)으로 묶어 관리합니다.

### 5.1 가상 노드 생성

> **👤 사용자 입력 예시**
> ```
> "거실_존"이라는 이름의 가상 노드를 생성해줘.
> 설명: "거실에 있는 모든 장치를 통합 관리"
> 포함 장치: 거실_조명, 거실_온도센서, 거실_에어컨, 거실_블라인드
> ```

> **🤖 AI 응답**
> ```
> ✅ 가상 노드가 생성되었습니다!
> ID: vn_20250703_거실_존
> 별명: 거실_존
> 포함 장치: 4개
>   - 거실_조명 (tasmota_light)
>   - 거실_온도센서 (tasmota_sensor)
>   - 거실_에어컨 (tasmota_ac)
>   - 거실_블라인드 (tasmota_blind)
> 
> 💡 이제 "거실_존의 모든 조명 꺼줘"라고 명령할 수 있습니다.
> ```

### 5.2 가상 노드에 장치 추가/제거

> **👤 사용자 입력 (추가)**
> ```
> 거실_환풍기를 거실_존 가상 노드에 추가해줘
> ```

> **👤 사용자 입력 (제거)**
> ```
> 거실_블라인드 장치를 거실_존에서 해제해줘
> ```

> **🤖 AI 응답**
> ```
> ✅ 거실_환풍기가 거실_존에 추가되었습니다.
> 현재 소속 장치: 거실_조명, 거실_온도센서, 거실_에어컨, 거실_환풍기 (총 4개)
> ```

### 5.3 가상 노드 삭제

> **👤 사용자 입력**
> ```
> 거실_존 가상 노드를 삭제해줘
> ```

> ⚠️ **주의**: 가상 노드 삭제 시 소속 장치들은 자동으로 분리되며, 개별 장치로 운영됩니다.

## 🔗 6. 가상 노드 + 시나리오 연동 (다중 장치 자동화)

가상 노드에 시나리오를 적용하면, 노드에 속한 장치들끼리 자동으로 연동됩니다.

> **👤 사용자 입력 예시**
> ```
> 시나리오 온습도_환기_제어를 거실_존 가상 노드에 적용해줘.
> 
> 역할 매핑:
>   temperature_sensor: 거실_온도센서
>   humidity_sensor: 거실_온도센서
>   living_room_light: 거실_조명_POWER1
>   fan_actuator: 거실_환풍기_POWER1
> 
> 파라미터: temp_max=28, humi_max=65
> 실행 모드: toggle
> 규칙 유형: 물리 규칙으로 각 장치에 직접 설치해줘
> ```

> **🤖 AI 응답**
> ```
> ✅ 시나리오 적용 완료! (물리 규칙)
> 
> 매핑 결과:
>   temperature_sensor → 거실_온도센서 (tele#temperature)
>   humidity_sensor → 거실_온도센서 (tele#humidity)
>   living_room_light → 거실_조명 (POWER1)
>   fan_actuator → 거실_환풍기 (POWER1)
> 
> 각 장치에 설치된 물리 규칙:
> 
> [거실_온도센서] (센서/트리거 장치)
>   Rule1: ON tele#temperature>=28 AND tele#humidity>=65 DO PUBLISH cmnd/거실_조명/POWER1 ON ENDON
>   Rule2: ON tele#temperature<28 OR tele#humidity<65 DO PUBLISH cmnd/거실_조명/POWER1 OFF ENDON
> 
> [거실_환풍기] (액추에이터 장치)
>   Rule1: ON tele#temperature>=28 AND tele#humidity>=65 DO POWER1 ON ENDON
>   Rule2: ON tele#temperature<28 OR tele#humidity<65 DO POWER1 OFF ENDON
> 
> ⚠️ 물리 규칙은 각 장치가 독립적으로 동작하므로, 서버 장애에도 영향을 받지 않습니다.
> ```

## 📐 7. DSL 구조 상세 해설 (개발자 참고)

AI가 내부적으로 생성하는 DSL(JSON) 구조를 이해하면 더 정교한 자동화를 요청할 수 있습니다.

### 7.1 시스템 규칙 DSL (서버 기반)

```json
{
  "rule_type": "system",
```

### 7.2 물리 규칙 DSL (Tasmota Rule)

```json
{
  "rule_type": "physical",
  "target_device": "환풍기",
  "rule_number": 1,
  "trigger": "tele#temperature>=28 AND tele#humidity>=65",
  "action": "POWER1 ON",
  "tasmota_syntax": "ON tele#temperature>=28 AND tele#humidity>=65 DO POWER1 ON ENDON"
}
```

### 7.3 시나리오 템플릿 DSL (toggle 모드)

```json
{
  "scenario_id": "sc_20250703_a1b2c3",
  "name": "온습도_환기_제어",
  "domain": "smart_home",
  "device_roles": [
    { "role": "temperature_sensor" },
    { "role": "humidity_sensor" },
    { "role": "living_room_light" },
    { "role": "fan_actuator" }
  ],
  "parameters": {
    "temp_max": { "type": "number", "default": 28 },
    "humi_max": { "type": "number", "default": 60 }
  },
  "execution_mode": "toggle",
  "steps": [
    {
      "name": "control_step",
      "conditions": [
        {
          "and": [
            { "role": "temperature_sensor", "field": "temperature", "operator": ">=", "value": "$temp_max" },
            { "role": "humidity_sensor", "field": "humidity", "operator": "<=", "value": "$humi_max" }
          ]
        }
      ],
      "action_true": {
        "steps": [
          { "role": "living_room_light", "command": "Power", "payload": "ON" },
          { "role": "fan_actuator", "command": "Power", "payload": "ON" }
        ]
      },
      "action_false": {
        "steps": [
          { "role": "living_room_light", "command": "Power", "payload": "OFF" },
          { "role": "fan_actuator", "command": "Power", "payload": "OFF" }
        ]
      }
    }
  ]
}
```

### 7.4 규칙 유형 자동 변환 로직

AI는 다음 기준으로 규칙 유형을 자동 판단합니다.

| 판단 기준 | 시스템 규칙 | 물리 규칙 |
|-----------|------------|-----------|
| **장치 개수** | 2개 이상 연동 | 단일 장치만 사용 |
| **조건 복잡도** | 시간대, 요일, 다중 AND/OR | 기본 비교 연산자만 |
| **외부 API 호출** | 필요 시 | 불가능 |
| **응답 속도 요구** | 느려도 무방 | 밀리초 단위 필요 |
| **인터넷 연결** | 항상 온라인 | 불안정/오프라인 환경 |

## ⚙️ 9. 문제 해결 (Troubleshooting)

| 상황 | 확인/해결 방법 |
|:---|:---|
| **시스템 규칙이 동작하지 않음** | `"생성된 규칙 리스트 보여줘"` → 활성 상태 및 쿨다운 확인, 서버 연결 상태 확인 |
| **물리 규칙이 동작하지 않음** | `"[장치]의 Rule 상태 보여줘"` → Rule이 활성화(ON) 상태인지 확인, Tasmota Console에서 직접 테스트 |
| **센서 값이 안 바뀜** | `"[장치]의 Telemetry 주기 10초로 변경해줘"` 또는 `"TelePeriod 10"` 명령어 사용 |
| **가상 노드에 장치가 없음** | `"현재 시스템에 등록된 전체 기기 목록이랑 가상 노드 보여줘"` |
| **물리 규칙 충돌** | `"[장치]의 Rule1, Rule2, Rule3 삭제해줘"` → 충돌하는 Rule 정리 후 재생성 |
| **MQTT 명령 실패** | `"[장치] 온라인 상태 확인해줘"` → 장치 IP/Ping 체크 |
| **너무 많은 규칙 생성됨** | `"모든 규칙 삭제해줘"` (주의: 시스템 규칙 + 물리 규칙 모두 삭제) |
| **규칙 유형 혼동** | `"이 규칙은 시스템 규칙인가요 물리 규칙인가요?"` → AI가 현재 규칙 유형 설명 |

## 📚 10. 규칙 유형별 명령어 요약표

| 목적 | 시스템 규칙 명령어 | 물리 규칙 명령어 |
|:---|:---|:---|
| **생성** | *"[장치]의 [센서]가 [값]보다 크면 [채널] ON 하는 시스템 규칙 만들어줘"* | *"[장치]에 물리 규칙으로 [센서]>= [값] → [채널] ON 설정해줘"* |
| **변환** | *"이 규칙을 물리 규칙으로 변환해줘"* | *"이 규칙을 시스템 규칙으로 변환해줘"* |
| **조회** | *"시스템 규칙 목록 보여줘"* | *"[장치]의 물리 규칙 목록 보여줘"* |
| **활성화** | *"시스템 규칙 [ID] 활성화해줘"* | *"Rule1 ON"* (Tasmota Console) |
| **비활성화** | *"시스템 규칙 [ID] 비활성화해줘"* | *"Rule1 OFF"* (Tasmota Console) |
| **삭제** | *"시스템 규칙 [ID] 삭제해줘"* | *"[장치]의 Rule1 삭제해줘"* |

## 🎯 11. 마무리

이 매뉴얼의 핵심은 **"특정 장치 이름을 몰라도, 역할과 동작만 자연어로 설명하면 AI가 알아서 처리해준다"**는 것입니다.

### 🚀 빠른 시작 가이드

1. **처음에는** `"[내 장치] 리스트 보여줘"` 로 자신의 장치를 확인하세요.
2. **그 다음** 원하는 자동화를 대화하듯이 설명하세요.
3. **규칙 유형 선택**: 
   - 복잡한 연동 → **시스템 규칙**
   - 빠른 응답/오프라인 대비 → **물리 규칙**
4. **복잡한 설정**은 가상 노드와 시나리오 템플릿을 조합하여 간결하게 관리하세요.

### 💡 핵심 포인트

- **시스템 규칙** = 서버에서 실행 (복잡한 조건, 다중 장치)
- **물리 규칙** = 장치에서 실행 (빠름, 인터넷 불필요)
- **물리 규칙은 Tasmota Rule** = "물리 룰" 이라고도 불림
- 규칙 생성 시 **규칙 유형을 명시**하면 더 정확한 결과를 얻을 수 있음
- 명시하지 않으면 AI가 **자동으로 적절한 유형을 추천**

AI가 모든 기술적 세부사항(DSL, MQTT, Rule 배포, 유형 변환)을 내부에서 처리하므로, 사용자는 **무엇을 할지(What)**에만 집중하면 됩니다. 😊
